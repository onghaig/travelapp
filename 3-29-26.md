# Travvy — Feature Set & UI Specification

You are adding features to an existing travel planning app called **Travvy**. The AI layer uses **OpenAI's API** (`gpt-4o` for the orchestrator, `gpt-4o-mini` for sub-agents). Do not change the existing project structure — extend it.

---

## AI Model Usage

| Agent | Model | Why |
|---|---|---|
| Orchestrator | `gpt-4o` | Complex reasoning, trip planning decisions |
| Flight, Lodging, Events | `gpt-4o-mini` | Fast parallel search, structured output |
| Budget, Calendar, Verification | `gpt-4o-mini` | Math and formatting tasks |
| Booking | `gpt-4o` | Needs to reason about form-filling strategy |

All agents use OpenAI function calling (`tools` parameter). Run flight, lodging, and event searches in parallel via `Promise.all()` (JS) or `asyncio.gather()` (Python).

---

## Feature Set

### Feature 1 — Progressive Onboarding Chat

The app opens to a full-screen chat. The AI greets the user with *"Where do you want to go?"* and collects all of the following through natural conversation — never in a form, never all at once:

- Destination(s)
- Departure and return dates
- Number of travelers
- Preferred airlines (or no preference)
- Preferred flight times (morning / afternoon / evening / any)
- Accommodation preference (Airbnb / hotel / both)
- Total budget with optional per-category split (flights, lodging, activities)
- Planned activities or events (concerts, restaurants, specific sights)
- Any special requirements (accessibility, dietary, etc.)

**Rules:**
- Never re-ask for information already given
- Store everything in a persistent `tripState` object as it is collected
- Auto-fill all downstream fields from `tripState` — the user should never type their name or dates twice
- Once destination + dates + travelers + budget are known, trigger all three searches immediately without waiting for more info

---

### Feature 2 — Multi-Agent Search (Parallel)

When search is triggered, run these three simultaneously:

1. **Flight Agent** — searches via mock Amadeus (Phase 1) or real Amadeus API (Phase 2). Returns top 4 flights sorted by price, each with: airline, flight number, departure/arrival times, duration, stops, total price, price per person.

2. **Lodging Agent** — searches hotels and Airbnbs. Returns top 4 options sorted by price, each with: name, type, neighborhood, price per night, total price, rating, review count, amenities list.

3. **Events Agent** — finds activities at the destination. Returns up to 6 options, each with: name, category, price, duration, rating, whether booking is required.

After all three return, immediately invoke the **Budget Agent** to validate the combined costs against the user's budget.

### Feature 3 — Budget Tracking (Live)

A **Budget Agent** runs as a persistent watchdog. It is called:
- After every search round
- After the user selects any option
- After any mid-process change request

It returns:
```json
{
  "total_estimated": 3240,
  "budget_total": 4000,
  "remaining": 760,
  "over_budget": false,
  "breakdown": [
    { "category": "flights", "cost": 1800, "percentage": 45 },
    { "category": "lodging", "cost": 1200, "percentage": 30 },
    { "category": "events", "cost": 240, "percentage": 6 }
  ],
  "warning": null
}
```

If `over_budget` is true, the orchestrator must proactively suggest cheaper alternatives before proceeding.

---

### Feature 4 — Calendar Sync

A **Calendar Agent** builds the trip timeline. It is called:
- After the user selects a flight (adds departure + return events)
- After the user selects lodging (adds check-in + check-out)
- After each event/activity is added

Each calendar event has:
```json
{
  "type": "flight | lodging | event",
  "title": "JAL Flight JL006 — JFK → NRT",
  "start_time": "2025-06-01T10:30:00",
  "end_time": "2025-06-02T14:45:00",
  "location": "Narita International Airport",
  "color": "#3B82F6"
}
```

Color coding: flights = `#3B82F6` (blue), lodging = `#10B981` (green), events = `#8B5CF6` (purple).

Optional: Google Calendar export button — generates an `.ics` file from the events array.

---

### Feature 5 — Pre-Booking Verification

Before any booking begins, a **Verification Agent** reviews the full plan and returns:
```json
{
  "verified": true,
  "issues": [],
  "summary": "10-day Tokyo trip, 2 travelers, $3,240 total, within $4,000 budget.",
  "ready_to_book": true
}
```

If `issues` is non-empty (e.g. a flight arrives after hotel check-in, or budget is exceeded), surface them to the user and block booking until resolved.

The orchestrator then presents the full itinerary and asks: *"Does everything look correct? You can ask for any changes below."*

---

### Feature 6 — Automated Booking Flow

A **Booking Agent** handles bookings sequentially: flights → lodging → events.

For each item:
1. Try **Playwright** — spin up a headless browser, navigate to the booking site, fill all fields from `tripState`, reach the checkout page, take a screenshot, stop.
2. If Playwright fails or times out (30s), fall back to a **deep link** — construct a pre-filled URL with all search params and open it in a new tab.

The booking agent returns:
```json
{
  "status": "at_checkout | deep_link",
  "checkout_url": "https://...",
  "screenshot_b64": "...",
  "prefilled": true,
  "message": "We've filled in your details. Complete your payment below."
}
```

**Important:** The booking agent ALWAYS stops before completing payment. The user manually completes the transaction. After confirming, they click "I've completed this booking" and the app marks it confirmed and moves to the next item.

---

### Feature 7 — Mid-Process Change Requests

A floating chat input (bottom-right corner, every screen) accepts plain-English changes at any point:

- *"Find a cheaper flight"* → re-runs Flight Agent only
- *"Switch to a hotel instead of Airbnb"* → re-runs Lodging Agent only
- *"Add a day trip to Mount Fuji"* → re-runs Events Agent, updates Calendar Agent and Budget Agent
- *"Push everything back a day"* → re-runs all three agents with new dates

The orchestrator identifies which agents are affected and re-runs only those. It then diffs the old vs. new plan and presents only what changed.

---

### Feature 8 — Trip Persistence

All trip state persists across sessions:
- Active state in Redis (keyed by `trip_id`)
- Full trip data in Supabase (`trips`, `bookings`, `calendar_events`, `budget_items`, `messages` tables)
- Returning users see their saved trip on login, exactly where they left off

## UI Elements

### Screen 1 — Onboarding Chat (Home)

**Layout:** Full-screen, vertically centered on first load. Transitions to a split layout once trip state begins filling in.

**Components:**

- **Hero greeting** — large display text: *"Where do you want to go?"* in a serif or editorial font. Fades out once conversation starts.
- **Message bubbles** — user messages right-aligned with an accent background. Assistant messages left-aligned in a card with subtle border. Streaming text renders token by token with a blinking cursor.
- **Typing indicator** — three animated dots while the AI is thinking or tools are running.
- **Tool status bar** — a slim bar below the last message showing which agent is currently running: *"Searching flights..."*, *"Finding hotels..."*. Disappears when done.
- **Trip state sidebar** (appears after first answer) — a right-side panel that populates in real time as the user answers questions. Shows: destination chip, date range, traveler count, budget bar. Each field animates in as it is collected.
- **Input bar** — fixed at the bottom. Text input + send button. Disabled while the AI is responding.

---

### Screen 2 — Planning Dashboard

**Layout:** Two-panel. Left 60% is the calendar. Right 40% is the budget tracker and option cards stacked vertically.

**Left Panel — Calendar:**

- Month view by default, toggle to week view in the top-right
- Events render as colored chips on their date cells, clipped to one line with a tooltip on hover showing full details
- Clicking a chip opens a detail drawer from the right with the full event info and an option to remove it
- Empty days show a faint `+` on hover — clicking opens the FloatingChat pre-filled with *"Add an event on [date]"*
- Navigation: previous/next month arrows, "Today" button

**Right Panel — Budget Tracker:**

- **Ring chart** at the top — donut chart showing total spent vs budget. Segments colored by category. Center shows percentage used. Turns amber at 80%, red at 100%.
- **Category breakdown** below the ring — four rows (flights, lodging, events, misc) each with a label, amount, and a thin progress bar.
- **Budget limit input** — editable field showing the total budget. User can update it inline; the ring and bars recompute immediately.
- **Option cards** below the budget — three card groups: Flights, Lodging, Events. Each card shows the key details and a "Select" button. Selecting a card:
  - Highlights it with an accent border
  - Deselects others in the group
  - Triggers a Budget Agent update
  - Adds the item to the calendar

---

### Screen 3 — Confirmation

**Layout:** Single-column, full-width, centered max-width container.

**Components:**

- **Calendar strip** — a compact horizontal timeline at the top showing all events across the trip duration, color-coded
- **Itinerary list** — every booking as a card: type icon (✈ 🏨 🎯), provider name, exact times, location, cost. Cards are read-only but have an "Edit" button that opens the FloatingChat pre-filled with that item's details
- **Traveler info section** — pre-filled from onboarding. Editable fields for: full name, email, phone. Clearly labeled *"This info will be used to pre-fill your bookings."*
- **Cost summary card** — total estimated cost, budget limit, remaining budget, and a green/red indicator
- **Orchestrator message** — a distinct AI message card above the CTA: *"Does everything look correct? You can ask for any changes below."*
- **CTA button** — full-width: *"Confirm & Start Booking"*. Disabled if `verified: false` from the Verification Agent.

---

### Screen 4 — Booking Flow

**Layout:** Centered modal-style progress flow over a dimmed dashboard.

**Components:**

- **Step indicator** — top of modal: Flights → Lodging → Events, with the current step highlighted
- **Booking card** — shows what is being booked: provider, dates, cost, traveler name
- **Status area:**
  - Loading state: spinner + *"Opening booking page..."*
  - Playwright success: screenshot of the checkout page in an `<iframe>`, message: *"We've filled in your details. Complete your payment on the page below."*
  - Deep link fallback: a prominent button *"Open Booking Page →"* that opens the prefilled URL in a new tab, message: *"Complete your booking on [Provider], then return here to confirm."*
- **Confirm button** — *"I've completed this booking ✓"*. Marks the item confirmed in Supabase, moves to the next step.
- **Skip button** — *"Skip for now"* in muted text below. Marks the item as pending and moves on.

---

### Screen 5 — My Trip Tab

**Layout:** Full-width list with a sticky summary header.

**Components:**

- **Summary header** — destination, dates, total spent vs budget, number of confirmed bookings
- **Booking cards** — one per booking, each showing:
  - Type icon + provider name
  - Dates and times
  - Cost (estimated if pending, actual if confirmed)
  - Status badge: `Confirmed` (green), `Pending` (amber), `Cancelled` (red strikethrough)
  - Confirmation number (if confirmed)
  - *"Manage Booking →"* link that opens the provider's manage page in a new tab
- **Add item button** — opens FloatingChat pre-filled with *"Add a booking to my trip"*

---

### Global Component — FloatingChat

Present on every screen after onboarding.

- Collapsed state: a circular chat icon fixed to the bottom-right corner with a subtle pulse animation when the AI has something to say
- Expanded state: a compact chat panel (320px wide, 400px tall) anchored to the bottom-right
- Header: *"Ask for changes..."* with a collapse button
- Same streaming chat interface as the onboarding screen
- Sends to the change-request endpoint, not the main chat endpoint
- After a change is processed, shows a diff summary: *"Updated: Flight changed to morning departure, saving $120. Budget updated."*

## SSE Chunk Types

The backend streams these event types. The frontend must handle all of them:

```typescript
type ChatChunk =
  | { type: 'text_delta';        content: string }
  | { type: 'tool_start';        tool: string; status: string }
  | { type: 'tool_result';       tool: string; data: Record<string, unknown> }
  | { type: 'trip_state_update'; state: TripState }
  | { type: 'calendar_update';   events: CalendarEvent[] }
  | { type: 'budget_update';     breakdown: BudgetBreakdown }
  | { type: 'done' }
```

On `tool_start` — show the tool status bar with the status message.
On `tool_result` — populate the relevant option cards (flights/lodging/events).
On `calendar_update` — re-render the calendar.
On `budget_update` — re-render the ring chart and breakdown rows.
On `trip_state_update` — update the trip state sidebar on the chat screen.
On `done` — hide the typing indicator and tool status bar.
